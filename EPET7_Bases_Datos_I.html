¿Por qué existen las bases de datos?

Del fichero de cartón al sistema relacional

Antes de las bases de datos, las empresas guardaban su información en ficheros físicos: cajones, carpetas y fichas de cartón. El tiempo para manipular esos datos era enorme. La informática reprodujo ese modelo —seguimos hablando de archivos, carpetas y formularios— pero lo hizo millones de veces más eficiente.

Archivos tradicionales vs. Bases de datos relacionales

⚠ Sistema de archivos tradicional
  • Datos redundantes: la misma información se repite en múltiples archivos (nombre y dirección del cliente en CLIENTES y en FACTURAS).
  • Costo de almacenamiento elevado al guardar el mismo dato en varias aplicaciones.
  • Tiempos de procesamiento elevados por la falta de optimización.
  • Alta probabilidad de inconsistencia: el mismo cliente puede aparecer como "Juan" en un archivo y "Joaquín" en otro.
  • Difícil modificación: cada cambio debe replicarse en todas las copias del dato, algo casi imposible en la práctica.
  • Cada programa maneja sus propios datos de forma caótica e independiente.
✔ Base de datos relacional
  • Menor redundancia: los datos se almacenan una sola vez y se referencian desde cualquier aplicación.
  • Menor espacio de almacenamiento gracias a una mejor estructuración de los datos.
  • Acceso más eficiente: la organización produce un resultado óptimo en rendimiento.
  • Datos documentados gracias a los metadatos que describen la estructura de la BD.
  • Independencia entre datos y programas: se pueden modificar los datos sin tocar el código de las aplicaciones.
  • Integridad y mayor seguridad: acceso controlado por usuario y consistencia garantizada.
Evolución histórica

Modelos de bases de datos

Las bases de datos evolucionaron junto con la tecnología. Cada modelo resolvió las limitaciones del anterior. Hoy el modelo relacional domina, pero convive con los demás en casos específicos.

MODELO 01 — Origen
Jerárquico

Las relaciones entre registros forman una estructura en árbol: cada nodo tiene un padre y puede tener muchos hijos, pero no al revés. Fue el modelo de las primeras bases de datos.

Ejemplos: IMS de IBM — Registro de Windows
MODELO 02 — 1969
En red

Admite relaciones más complejas que el jerárquico: cada registro puede relacionarse con varios otros y seguirse por distintos caminos. Inventado por Charles Bachman, estandarizado por CODASYL.

Estándar: CODASYL (1969)
MODELO 03 — El dominante
Relacional ★

Almacena los datos en tablas (filas y columnas). Las tablas se conectan entre sí mediante claves comunes. Es el modelo más extendido en el mundo y el que estudiamos en este curso.

MySQL — PostgreSQL — Oracle — SQL Server — SQLite
MODELO 04 — Analítico
Multidimensional

Similar al relacional pero con N dimensiones en lugar de las dos filas-columnas. Ofrece el aspecto de una hoja de cálculo tridimensional. Usado en análisis de grandes volúmenes de datos (Business Intelligence).

Ejemplo: cubos OLAP — Data Warehouses
MODELO 05 — Moderno
Orientado a objetos

Diseñado siguiendo el paradigma OO. Soporta tipos de datos complejos como gráficos, imágenes, voz y texto de forma natural. Muy usado en aplicaciones web multimedia.

Ejemplo: MongoDB — CouchDB — db4o
Vocabulario fundamental

Conceptos básicos de una base de datos

Estos son los términos que aparecen en toda discusión sobre bases de datos. Dominarlos es el primer paso para diseñar y consultar cualquier sistema.

📊
Dato

Hecho conocido que puede registrarse y que tiene un significado implícito. El significado depende del contexto: "fecha" en una BD de ventas es la fecha de factura; en una BD de música, la fecha de grabación.

Ejemplo: 3256789 | "Elena Sánchez" | "18 de julio de 1880"
🏷️
Entidad

Todo aquello de lo cual interesa guardar datos. Puede ser una persona, un objeto, un evento o un concepto del mundo real que tenga existencia independiente.

Ejemplos: CLIENTES | FACTURAS | PRODUCTOS
📋
Tabla

Conjunto de filas y columnas bajo un mismo nombre que representa los valores almacenados para una entidad. Cada tabla tiene un nombre único en la base de datos.

Ejemplo: tabla CLIENTES con todos los clientes de la empresa
📌
Campo (atributo)

Cada columna de una tabla. Identifica una familia de datos del mismo tipo. Tiene un nombre, un tipo de dato y una longitud máxima definidos en los metadatos.

Ejemplo: campo "fechaNacimiento" en tabla CLIENTES
📄
Registro (tupla)

Cada fila de la tabla. Representa una instancia concreta de la entidad: un cliente, una factura, un producto específico con todos sus datos.

Ejemplo: {1225 | "Juan García" | "Guaná 1202" | ...}
🔑
Clave primaria (PK)

Campo o conjunto de campos que identifica de forma única a cada registro de la tabla. No puede repetirse ni estar vacío. Es el identificador irrepetible de cada fila.

Ejemplo: CódigoCliente en CLIENTES — NúmeroFactura en FACTURAS
🔗
Clave foránea (FK)

Campo de una tabla que contiene la clave primaria de otra tabla. Es el mecanismo que crea relaciones entre tablas. Garantiza la integridad referencial: no puede apuntar a un registro inexistente.

Ejemplo: CódigoCliente en FACTURAS apunta a CLIENTES
🗂️
Metadatos

Datos acerca de los datos. Describen la estructura de la base de datos: qué tipo tiene cada campo, cómo se llama, cómo están agrupados y cómo se relacionan. Son el "manual de instrucciones" de la BD.

Ejemplo: Num → Numérico → longitud 4 | Nombre → Alfab. → 20
🔍
Consulta (Query)

Instrucción escrita en SQL que hace una petición a la base de datos. Puede recuperar, insertar, modificar o eliminar datos. Es la forma principal de interactuar con la BD.

SELECT * FROM CLIENTES WHERE ciudad = 'Jardín América'
👁️
Vista

Tabla virtual obtenida al guardar una consulta. No almacena datos físicamente; muestra el resultado de la consulta como si fuera una tabla real. Se actualiza automáticamente con los datos subyacentes.

CREATE VIEW clientes_activos AS SELECT ...
📑
Informe

Listado ordenado de campos y registros seleccionados en un formato fácil de leer e imprimir. Permite presentar la información de la BD de forma estructurada y comprensible para el usuario final.

Ejemplo: facturas impagas de enero ordenadas por cliente
⚙️
Procedimiento almacenado

Script especial almacenado dentro de la BD que forma parte de su esquema. Ejecuta operaciones avanzadas o de mantenimiento de forma automatizada y repetible.

CREATE PROCEDURE actualizar_precios() ...

Claves primarias (PK)

CodCli identifica a cada cliente de forma única. NroFactura a cada factura. CodProd a cada producto. Ningún valor puede repetirse ni estar vacío.

Claves foráneas (FK)

CodCli en FACTURAS apunta a CLIENTES. CodProd en FACTURAS apunta a PRODUCTOS. Si el cliente 1225 no existe en CLIENTES, la factura F0001 no puede crearse.

Integridad referencial

El SGBD garantiza que ninguna FK apunte a un registro inexistente. Si intentás borrar el cliente 1225 mientras tiene facturas, el sistema lo impide automáticamente.

Sistemas gestores

¿Qué es un SGBD y cuáles existen?

Un SGBD (Sistema Gestor de Bases de Datos) es el software que permite definir, crear, mantener y consultar una base de datos con acceso controlado. Proporciona DDL (definición), DML (manipulación) y DCL (control de acceso).

DDL — Data Definition Language

Define y modifica la estructura de la BD: crea tablas, define tipos de datos y establece restricciones.

CREATE TABLE | ALTER TABLE | DROP TABLE
DML — Data Manipulation Language

Inserta, actualiza, elimina y consulta los datos almacenados en la BD.

SELECT | INSERT | UPDATE | DELETE
DCL — Data Control Language

Controla los permisos de acceso: quién puede ver, modificar o administrar la BD.

GRANT | REVOKE | DENY

SGBD más conocidos

🐬
MySQL
Relacional — Gratuito / Open Source

El más usado en aplicaciones web del mundo. Base de sitios como WordPress, Facebook (en sus orígenes) y millones de sistemas empresariales.

🐘
PostgreSQL
Relacional — Gratuito / Open Source

Preferido para sistemas que requieren alta confiabilidad y funciones avanzadas. Soporta JSON, arrays y tipos de datos complejos.

🔶
Oracle
Relacional — Corporativo / De pago

Uno de los más potentes y usados en grandes empresas y bancos. Versión gratuita disponible con fines educativos (Oracle XE).

🪟
Microsoft Access
Relacional — Ofimático

SGBD para bases de datos pequeñas. Ideal para doméstico y pequeñas empresas. Incluye interfaz visual para crear tablas y formularios sin escribir SQL.

📦
SQLite
Relacional — Embebido / Gratuito

BD que vive dentro de un único archivo. Usado en apps móviles (Android, iOS), navegadores web y herramientas de escritorio. Sin servidor.

🍃
LibreOffice Base
Ofimático — Gratuito / Open Source

Alternativa libre a Microsoft Access. Permite crear bases de datos con interfaz gráfica. Ideal para aprendizaje en el aula sin costo de licencia.
