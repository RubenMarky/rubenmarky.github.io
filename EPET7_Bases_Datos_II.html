Unidad 2 — Diseño conceptual

Modelo Entidad-Relación (MER)

El MER es el modelo más usado para el diseño conceptual de bases de datos. Introducido por Peter Chen en 1976, se basa en entidades, atributos y relaciones.

Entidad fuerte

Entidad que tiene existencia propia e independiente. Se identifica con su propia clave primaria sin depender de otra entidad.

Ejemplo: CLIENTE, PRODUCTO, EMPLEADO.

Entidad débil

Entidad que depende de una fuerte para existir o identificarse. En dependencia de identificación, su PK se forma con la PK de la fuerte + su discriminador.

Ejemplo: MUNICIPIO depende de PROVINCIA.

Cardinalidad 1:1

A cada elemento de A le corresponde como máximo un elemento de B y viceversa.

Ejemplo: Una persona tiene un único DNI. Un DNI pertenece a una única persona.

Cardinalidad 1:N

Un elemento de A puede relacionarse con muchos de B, pero cada B solo se relaciona con un A.

Ejemplo: Un departamento tiene muchos empleados. Cada empleado pertenece a un departamento.

Cardinalidad N:M

Cualquier cantidad de A puede relacionarse con cualquier cantidad de B. Siempre genera una tabla intermedia en el modelo relacional.

Ejemplo: Un alumno cursa muchas materias. Una materia la cursan muchos alumnos.

Atributo propio de relación

Atributo cuyo valor solo existe en la relación, no en ninguna de las entidades por separado.

Ejemplo: En la relación Compra (CLIENTE–PRODUCTO), la Cantidad solo existe cuando un cliente compra un producto.

MER Extendido — Jerarquía

Relación "es un tipo de". La entidad padre se llama supertipo y las hijas, subtipos. Se representa con triángulo invertido. Puede ser total (todos los casos están cubiertos) o parcial.

Ejemplo: EMPLEADO → INGENIERO, SECRETARIO, TÉCNICO.

Paso al Modelo Relacional

Regla clave: Relación N:M → siempre genera tabla. Relación 1:N → la PK del lado "1" pasa como FK al lado "N". Relación 1:1 → la FK va a la entidad con participación mínima 0 para evitar nulos.

Unidad 2 — Formas normales

Normalización: 1FN, 2FN, 3FN y FNBC

La normalización es el proceso de organizar las tablas para eliminar redundancias e inconsistencias. Se aplica revisando las dependencias funcionales de cada tabla.

1FN
Primera Forma Normal
Requisito: todos los valores de cada campo deben ser atómicos (indivisibles).

No puede haber campos que contengan múltiples valores en una sola celda ni grupos repetidos. Si un campo almacena varios teléfonos, viola la 1FN.

❌ Alumno: {DNI, Nombre, Teléfonos}
✓ Solución: tabla separada TELÉFONOS (DNI, Teléfono)
2FN
Segunda Forma Normal
Requisito: estar en 1FN y que todos los atributos no clave dependan COMPLETAMENTE de la PK.

Solo se aplica cuando la PK es compuesta. Si un campo depende solo de una parte de la PK, hay que separarlo en otra tabla.

❌ MATRÍCULA(DNI, Curso, FechaMatrícula, Tutor)
Tutor depende solo de Curso, no de (DNI, Curso).
✓ Tabla CURSOS(Curso, Tutor) separada.
3FN
Tercera Forma Normal
Requisito: estar en 2FN y no existir dependencias transitivas.

Una dependencia transitiva ocurre cuando un atributo no-clave depende de otro atributo no-clave. Todos los atributos deben depender directamente de la PK.

❌ ALUMNO(DNI, Nombre, Localidad, Provincia)
DNI → Localidad → Provincia (transitiva)
✓ Tabla LOCALIDADES(Localidad, Provincia) separada.
FNBC
Forma Normal de Boyce-Codd
Requisito: estar en 3FN y no existir solapamiento de claves candidatas compuestas.

Solo aplica cuando hay varias claves candidatas compuestas que se solapan. Poco frecuente, pero importante en diseños complejos.

❌ SUMINISTROS(CIF, Nombre, CódigoPieza, Cantidad)
CIF↔Nombre crea solapamiento de claves.
✓ Separar en PROVEEDORES(CIF, Nombre) + SUMINISTROS(CIF, CódigoPieza, Cantidad).
Unidad 3 — Lenguaje de definición

DDL — Creación de tablas en MySQL

El DDL (Data Definition Language) define la estructura física de la base de datos. Las instrucciones más importantes son CREATE TABLE, ALTER TABLE y DROP TABLE.

Tipos de datos principales en MySQL
Texto

CHAR(n) — longitud fija. VARCHAR(n) — longitud variable hasta n. TEXT — texto largo. ENUM('v1','v2') — lista cerrada de valores.

Números

INT — entero. BIGINT — entero grande. DECIMAL(p,s) — decimal exacto. FLOAT / DOUBLE — punto flotante.

Fechas

DATE — solo fecha (YYYY-MM-DD). DATETIME — fecha y hora. TIMESTAMP — instante con zona horaria. YEAR — solo año.

Otros

BOOLEAN / TINYINT(1) — verdadero/falso. BLOB — datos binarios (imágenes). AUTO_INCREMENT — autonumérico.

Ejemplo completo: CREATE TABLE con todas las restricciones
Sistema de pedidos: tablas CLIENTES, PRODUCTOS y PEDIDOS con sus relaciones.
-- ====================================================
-- BASE DE DATOS: sistema_pedidos
-- ====================================================
CREATE DATABASE sistema_pedidos
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_spanish_ci;

USE sistema_pedidos;

-- Tabla CLIENTES (entidad principal)
CREATE TABLE CLIENTES (
  cod_cliente   INT          AUTO_INCREMENT,
  nombre        VARCHAR(60)  NOT NULL,
  apellido      VARCHAR(60)  NOT NULL,
  email         VARCHAR(100) UNIQUE,
  telefono      CHAR(15),
  fecha_alta    DATE         DEFAULT (CURDATE()),
  activo        BOOLEAN      DEFAULT TRUE,
  CONSTRAINT pk_clientes PRIMARY KEY (cod_cliente),
  CONSTRAINT chk_email   CHECK (email LIKE '%@%.%')
);

-- Tabla PRODUCTOS
CREATE TABLE PRODUCTOS (
  cod_producto  INT           AUTO_INCREMENT,
  descripcion   VARCHAR(100)  NOT NULL,
  precio        DECIMAL(10,2) NOT NULL,
  stock         INT           DEFAULT 0,
  categoria     ENUM('electrónica','ropa','alimentos'),
  CONSTRAINT pk_productos  PRIMARY KEY (cod_producto),
  CONSTRAINT chk_precio    CHECK (precio > 0),
  CONSTRAINT chk_stock     CHECK (stock >= 0)
);

-- Tabla PEDIDOS (relación N:M entre CLIENTES y PRODUCTOS)
CREATE TABLE PEDIDOS (
  nro_pedido    INT           AUTO_INCREMENT,
  cod_cliente   INT           NOT NULL,
  cod_producto  INT           NOT NULL,
  cantidad      INT           NOT NULL    DEFAULT 1,
  fecha_pedido  DATETIME      DEFAULT     NOW(),
  CONSTRAINT pk_pedidos     PRIMARY KEY (nro_pedido),
  CONSTRAINT fk_ped_cliente FOREIGN KEY (cod_cliente)
    REFERENCES CLIENTES(cod_cliente)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,
  CONSTRAINT fk_ped_prod    FOREIGN KEY (cod_producto)
    REFERENCES PRODUCTOS(cod_producto)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,
  CONSTRAINT chk_cantidad   CHECK (cantidad > 0)
);
Opciones de ON DELETE y ON UPDATE
RESTRICT

Impide borrar/modificar la fila padre si tiene filas hijas referenciándola. La opción más segura para proteger la integridad.

CASCADE

Borra/modifica automáticamente todas las filas hijas cuando se borra/modifica la fila padre. Útil para mantener consistencia.

SET NULL

Al borrar la fila padre, la FK en las filas hijas queda en NULL. Requiere que la columna FK admita nulos.

SET DEFAULT

Al borrar la fila padre, la FK adopta su valor DEFAULT. Poco común pero útil para categorías genéricas.

Unidad 4 — Consultas

DML — Consultas SELECT

El SELECT es la instrucción más poderosa de SQL. Puede incluir filtros, ordenamiento, agrupaciones, subconsultas y combinación de múltiples tablas.

Estructura completa del SELECT
SELECT   [DISTINCT] columnas | expresiones | funciones
FROM     tabla1 [JOIN tabla2 ON condicion]
WHERE    condicion_de_filas
GROUP BY columnas_de_agrupacion
HAVING   condicion_sobre_grupos
ORDER BY columna [ASC | DESC]
LIMIT    n;

-- ORDEN DE EJECUCIÓN (no es el orden de escritura):
-- 1. FROM  2. WHERE  3. GROUP BY  4. HAVING  5. SELECT  6. ORDER BY  7. LIMIT
Funciones de agregación — GROUP BY y HAVING
Las funciones de agregación calculan un valor a partir de varias filas. HAVING filtra sobre grupos (como el WHERE pero después de agrupar).
-- Cantidad de pedidos y total gastado por cada cliente
SELECT
  c.nombre,
  c.apellido,
  COUNT(p.nro_pedido)              AS cantidad_pedidos,
  SUM(p.cantidad * pr.precio)      AS total_gastado,
  AVG(p.cantidad * pr.precio)      AS promedio_por_pedido,
  MAX(p.fecha_pedido)              AS ultimo_pedido
FROM CLIENTES c
  JOIN PEDIDOS p   ON c.cod_cliente  = p.cod_cliente
  JOIN PRODUCTOS pr ON p.cod_producto = pr.cod_producto
GROUP BY c.cod_cliente, c.nombre, c.apellido
HAVING COUNT(p.nro_pedido) > 2     -- solo clientes con más de 2 pedidos
ORDER BY total_gastado DESC;
Subconsultas
Una subconsulta usa el resultado de un SELECT dentro de otro. Se puede colocar en WHERE, HAVING o FROM.
-- Clientes que gastaron más que el promedio general
SELECT nombre, apellido
FROM CLIENTES
WHERE cod_cliente IN (
  SELECT cod_cliente
  FROM PEDIDOS p JOIN PRODUCTOS pr ON p.cod_producto = pr.cod_producto
  GROUP BY cod_cliente
  HAVING SUM(p.cantidad * pr.precio) > (
    SELECT AVG(total)
    FROM (
      SELECT SUM(p2.cantidad * pr2.precio) AS total
      FROM PEDIDOS p2 JOIN PRODUCTOS pr2 ON p2.cod_producto = pr2.cod_producto
      GROUP BY p2.cod_cliente
    ) totales
  )
);
Unidad 4 — Combinación de tablas

Tipos de JOIN

Los JOINs combinan filas de dos o más tablas basándose en una condición. La elección del tipo de JOIN determina qué filas aparecen cuando no hay coincidencia.

Inner Join
INNER JOIN (JOIN)

Solo las filas que tienen coincidencia en AMBAS tablas. Si no hay coincidencia, la fila no aparece en ningún lado.

FROM A JOIN B ON A.id = B.id
Left Outer Join
LEFT JOIN

TODAS las filas de la tabla izquierda (A), más las coincidencias de B. Si B no tiene coincidencia, aparece NULL.

FROM A LEFT JOIN B ON A.id = B.id
Right Outer Join
RIGHT JOIN

TODAS las filas de la tabla derecha (B), más las coincidencias de A. Si A no tiene coincidencia, aparece NULL.

FROM A RIGHT JOIN B ON A.id = B.id
Full Outer Join
FULL JOIN

TODAS las filas de AMBAS tablas. Donde no hay coincidencia en el otro lado, aparece NULL.

FROM A FULL JOIN B ON A.id = B.id
Self Join
SELF JOIN

Una tabla se une consigo misma. Útil para relaciones jerárquicas dentro de la misma tabla (empleado-jefe, categoría-subcategoría).

FROM EMPLEADOS e JOIN EMPLEADOS j ON e.jefe_id = j.emp_id
Natural Join
NATURAL JOIN

Une automáticamente por todas las columnas con el mismo nombre en ambas tablas. Peligroso si hay columnas con nombres coincidentes inesperados.

FROM A NATURAL JOIN B
Ejemplo: múltiples JOINs en una sola consulta
-- Informe completo de pedidos con nombre de cliente y producto
SELECT
  p.nro_pedido,
  CONCAT(c.nombre, ' ', c.apellido)  AS cliente,
  pr.descripcion                       AS producto,
  p.cantidad,
  pr.precio,
  (p.cantidad * pr.precio)             AS subtotal,
  p.fecha_pedido
FROM PEDIDOS p
  JOIN CLIENTES  c  ON p.cod_cliente  = c.cod_cliente
  JOIN PRODUCTOS pr ON p.cod_producto = pr.cod_producto
WHERE p.fecha_pedido >= '2026-01-01'
ORDER BY p.fecha_pedido DESC;
Unidad 5 — Modificación de datos

DML — INSERT, UPDATE y DELETE

Las instrucciones de modificación alteran los datos almacenados. Siempre deben usarse con cuidado, especialmente DELETE y UPDATE sin WHERE.

INSERT — Insertar datos
-- Insertar un cliente completo
INSERT INTO CLIENTES (nombre, apellido, email, telefono)
VALUES ('Juan', 'García', 'jgarcia@mail.com', '3755001234');

-- Insertar varios registros en una sola instrucción
INSERT INTO PRODUCTOS (descripcion, precio, stock, categoria)
VALUES
  ('Notebook 15"',   185000.00, 10, 'electrónica'),
  ('Mouse inalámbrico', 8500.00, 50, 'electrónica'),
  ('Remera talle M',   3200.00, 30, 'ropa');

-- Insertar datos obtenidos de una consulta
INSERT INTO CLIENTES_VIP (cod_cliente, nombre, apellido)
SELECT cod_cliente, nombre, apellido
FROM CLIENTES
WHERE cod_cliente IN (
  SELECT cod_cliente FROM PEDIDOS
  GROUP BY cod_cliente HAVING COUNT(*) > 10
);
UPDATE — Modificar datos
-- Actualizar un campo específico con condición
UPDATE PRODUCTOS
SET precio = precio * 1.10   -- aumentar precio 10%
WHERE categoria = 'electrónica';

-- Actualizar con subconsulta
UPDATE CLIENTES
SET activo = FALSE
WHERE cod_cliente NOT IN (
  SELECT DISTINCT cod_cliente
  FROM PEDIDOS
  WHERE fecha_pedido >= DATE_SUB(CURDATE(), INTERVAL 1 YEAR)
);
-- ⚠ UPDATE sin WHERE modifica TODOS los registros
DELETE — Eliminar datos
-- Eliminar un registro específico
DELETE FROM PEDIDOS
WHERE nro_pedido = 15;

-- Eliminar con subconsulta
DELETE FROM CLIENTES
WHERE cod_cliente IN (
  SELECT cod_cliente FROM CLIENTES_INACTIVOS
);
-- ⚠ DELETE sin WHERE elimina TODOS los registros de la tabla
-- ⚠ Si hay FK con ON DELETE RESTRICT, no podrás borrar si hay hijos
Unidad 5 — Control de transacciones

Transacciones y propiedades ACID

Una transacción es un conjunto de operaciones que se ejecutan como una unidad: o todas ocurren, o ninguna. Garantizan la consistencia de los datos.

A
Atomicidad

La transacción es todo o nada. Si cualquier operación falla, se revierten todas las anteriores. No puede haber estados intermedios.

C
Consistencia

La BD pasa de un estado válido a otro estado válido. Todas las restricciones de integridad se mantienen antes y después.

I
Aislamiento

Las transacciones concurrentes no se interfieren entre sí. Cada transacción ve la BD como si fuera la única que está operando.

D
Durabilidad

Una vez confirmada (COMMIT), los cambios son permanentes aunque el sistema falle. Se garantiza mediante logs de transacciones.

COMMIT, ROLLBACK y SAVEPOINT
-- Inicio de transacción (en MySQL con autocommit desactivado)
START TRANSACTION;

-- Operaciones de la transacción
UPDATE PRODUCTOS SET stock = stock - 5
WHERE cod_producto = 1;

INSERT INTO PEDIDOS (cod_cliente, cod_producto, cantidad)
VALUES (3, 1, 5);

-- Punto de ruptura intermedio
SAVEPOINT despues_de_pedido;

UPDATE CLIENTES SET activo = TRUE
WHERE cod_cliente = 3;

-- Si algo falla, volver al savepoint (no deshace todo)
ROLLBACK TO SAVEPOINT despues_de_pedido;

-- Confirmar todos los cambios de forma permanente
COMMIT;

-- Si algo grave falla antes del COMMIT, deshacer todo
ROLLBACK;
Unidad 6 — Seguridad

DCL — Control de acceso y seguridad

El DCL (Data Control Language) gestiona quién puede hacer qué en la base de datos. Se basa en usuarios, privilegios y roles.

Gestión de usuarios y privilegios en MySQL
-- Crear un usuario nuevo
CREATE USER 'alumno1'@'localhost'
  IDENTIFIED BY 'Contraseña_2026!';

-- Dar privilegios específicos sobre una BD
GRANT SELECT, INSERT, UPDATE
  ON sistema_pedidos.*
  TO 'alumno1'@'localhost';

-- Dar todos los privilegios (administrador de esa BD)
GRANT ALL PRIVILEGES
  ON sistema_pedidos.*
  TO 'admin_bd'@'localhost';

-- Quitar un privilegio específico
REVOKE DELETE
  ON sistema_pedidos.*
  FROM 'alumno1'@'localhost';

-- Ver los privilegios de un usuario
SHOW GRANTS FOR 'alumno1'@'localhost';

-- Eliminar un usuario
DROP USER 'alumno1'@'localhost';
